# Ayue Observatory Lab Sections

> 阶段：第二步／次级板块主题。
>
> 状态：讨论稿。本文档在 `SITE_ARCHITECTURE_V2.md` 已确认的架构边界内，定义 Lab 的板块结构、主题、技术自由和首版优先级，不进入具体项目实现。

## 1. 第二部分目标

Lab 不是一个统一模板下的实验列表。它是多个性质不同的动态空间集合。

第二部分需要确定：

- 哪些内容值得成为 Lab 的长期板块。
- 每个板块提供怎样的使用体验。
- 每个板块允许使用怎样的动态技术和数据。
- 哪些平台能力可以共享。
- 哪些视觉、结构和交互必须由具体项目独立设计。
- 首版先验证哪些板块，哪些只保留未来入口。

## 2. Lab 总体信息架构

建议结构：

```text
Lab
├── Visual Systems
├── NAS Observatory
├── Interactive Works
└── Open Lab
```

这四个板块不是技术栈分类，而是按内容存在的理由分类：

- `Visual Systems`：技术本身就是被观察的对象。
- `NAS Observatory`：真实系统数据是被观察的对象。
- `Interactive Works`：用户通过操作完成一次体验或探索。
- `Open Lab`：对可靠开源项目进行有明确目的的研究、改造和再表达。

## 3. Lab 首页

### 3.1 职责

Lab 首页负责：

- 解释 Lab 与静态 Observatory 的区别。
- 展示四个板块及其当前状态。
- 提供当前运行项目和最近更新项目入口。
- 显示数据更新时间与平台状态，但不变成监控 dashboard。
- 提供返回 Observatory 的明确路径。

### 3.2 首屏原则

- 首屏可以有动态背景或环境视觉，但必须轻于具体项目。
- 不在首屏同时播放多个高成本实验。
- 不使用指标大卡片堆叠。
- 不把 GitHub 星数、技术栈 Logo 或性能数字当成主要内容。
- 首屏应表达“这里可以运行和操作”，而不是“这里用了很多技术”。

### 3.3 项目发现方式

首版只需要：

- 按板块进入。
- 当前项目入口。
- 最近更新入口。

暂不需要搜索、标签筛选和复杂排序。

## 4. Visual Systems

### 4.1 定位

Visual Systems 容纳以博客内容理解、发现或操作为目的的高级动态前端实验。技术表现可以很强，但必须服务明确内容目标；纯生成视觉、Shader 和粒子效果只属于可复用修饰能力，不自动构成独立项目。

它不是特效素材库，也不为了满足“动态元素”要求制造页面。关闭视觉修饰后，正式项目仍必须具备可理解、可使用的核心价值。

### 4.2 内容范围

- 内容地图、空间索引与可视化导航。
- Canvas、SVG 和 WebGL 实验。
- 服务真实内容的粒子、噪声、流体、场和几何系统。
- 交互排版、空间导航和滚动叙事。
- 鼠标、触控、键盘、声音等输入映射。
- 非交互的环境动画与数字装置。
- 浏览器渲染能力和性能边界实验。

### 4.3 技术自由

- 可以全屏或占据主要视口。
- 可以使用独立色彩、排版和交互语言。
- 可以使用 WebGL、Canvas、SVG、Web Audio 和 Workers。
- 可以完全不使用 Archive 卡片结构。
- 可以按项目选择框架或原生实现。

### 4.4 必须保留

- 返回 Lab 和 Observatory 的路径。
- 项目目的和操作说明。
- 暂停、重置或重新生成能力。
- reduced-motion 和低性能降级。
- 资源释放和离开页面后的停止行为。

### 4.5 与全局架构的关系

主要使用纯客户端动态能力。默认不需要 NAS 数据，也不需要服务端写操作。

可能需要的公共能力：

- 全屏实验容器。
- 项目级资源加载。
- 帧率和性能降级工具。
- 参数预设与本地状态。
- Canvas / WebGL 错误降级。

### 4.6 首批内容方向

- `VS–001 Archive Atlas／档案地图`：七个真实公开 Archive Objects 已构成可搜索、筛选和导航的时间空间；节点由构建期 MDX 索引生成。
- `VS–002`：预留给用户提供的动态实验，可独立设计，不继承 VS–001 模板；工程来源与技术基线见 `VS002_TECHNICAL_BASELINE.md`。
- Temporal Field 渲染器：仅保留为未公开视觉能力原型，不占用项目编号。

## 5. NAS Observatory

### 5.1 定位

NAS Observatory 将经过脱敏和聚合的 NAS / Cloud 数据转化为可观察的动态环境。

它不是运维后台，也不以故障告警和管理操作为目标。它关注系统如何随时间变化，以及真实基础设施如何被重新表达和理解。

### 5.2 内容范围

- 存储容量和增长趋势。
- 服务、容器或功能分组的状态变化。
- CPU、内存、温度、网络等聚合趋势。
- 备份、同步和任务事件计数。
- 系统层级、依赖和数据流。
- 当前、24 小时、7 天或更长周期的回放。
- Cloud 与 NAS 之间经过允许的数据关系。

### 5.3 表达原则

- 优先表现变化、关系和时间，而不是单个即时数字。
- 可以使用星图、地形、生态、轨道、流场或时间切片等视觉隐喻。
- 隐喻必须有稳定映射，不能只追求好看。
- 页面必须明确数据更新时间、聚合范围和过期状态。
- 不使用红绿灯式告警作为主要视觉语言。

### 5.4 数据边界

允许：

- 聚合比例、区间和趋势。
- 经别名处理的服务类别。
- 降采样的时间序列。
- 人工确认可以公开的系统关系。

禁止：

- 文件名和完整路径。
- 用户、令牌、内网地址、设备序列号。
- 原始容器配置和完整日志。
- 可用于管理或控制 NAS 的接口。

### 5.5 与全局架构的关系

主要使用周期数据动态能力：

- Collectors 按项目频率采集。
- Data Gateway 提供只读快照。
- 默认更新频率为每小时；只有项目部署在 NAS 本机并完成资源与安全评估时，才考虑真正实时或高频更新。
- 历史数据保存聚合版本。
- 前端处理 loading、empty、error 和 stale。

可能需要的公共能力：

- 时间范围和回放控制。
- 统一的来源、单位和更新时间显示。
- 历史数据窗口加载。
- 数据 schema 版本兼容。

### 5.6 首批内容方向

- NAS Constellation：服务和系统关系的动态星图。
- Storage Weather：容量、写入和备份变化形成的长期环境。
- System Layers：设备、服务、工具和工作流的可展开层级。

这些仍是候选，不在第二步确定具体字段和技术。

## 6. Interactive Works

### 6.1 定位

Interactive Works 容纳需要用户操作、选择、探索或计算才能成立的完整作品。

它与 Visual Systems 的区别是：Visual Systems 主要观察视觉系统本身；Interactive Works 主要完成一个可理解的交互目标。

### 6.2 内容范围

- 数据探索器和历史回放工具。
- 算法、数学和系统概念的可操作演示。
- 基于公开 NAS 快照的查询和组合体验。
- 本地运行的小型计算器、模拟器和生成工具。
- 浏览器端 AI、图像、声音或文本实验。
- 不需要服务端写权限的交互应用。

### 6.3 首版边界

- 所有公开项目匿名可访问。
- 不触发 NAS 写操作。
- 不执行任意用户代码。
- 不保存私人输入到服务端。
- 如需保存状态，首选 URL、浏览器本地存储或可导出的本地文件。

### 6.4 技术自由

- 每个作品拥有独立信息架构。
- 可以使用复杂表单、拖放、时间轴、地图或图形编辑。
- 可以使用 Web Workers、WASM 和浏览器端模型。
- 可以读取 Data Gateway 的公开数据。

### 6.5 与全局架构的关系

主要使用纯客户端动态或周期数据动态。受控服务端交互暂不进入首版。

可能需要的公共能力：

- 本地状态保存与恢复。
- 分享参数的 URL 编码。
- 长计算任务取消。
- Workers / WASM 生命周期管理。
- 用户输入的本地隐私说明。

### 6.6 首批内容方向

- NAS 历史状态回放器。
- 系统依赖探索器。
- 数据映射或视觉参数实验器。
- 一个完全在浏览器运行的小型实用工具。

## 7. Open Lab

### 7.1 定位

Open Lab 用于研究、改造和重新表达可靠的开源项目或开放技术，不是简单部署他人项目，也不是开源链接收藏页。

一个项目只有在产生明确的新观察、新交互、新数据映射或适合本站的技术改造时，才进入 Open Lab。

### 7.2 可接受形式

- 对开源可视化或生成艺术技术进行独立主题创作。
- 将可靠开源工具改造成只读、脱敏的 NAS 数据体验。
- 对浏览器图形、WASM、地图或音频项目进行小范围实验。
- 以 sandbox 运行与主站完全隔离的开源演示。
- 记录上游、许可证、改动和安全边界。

### 7.3 不接受

- 未修改的开源项目镜像。
- 只为增加功能数量而部署的工具集合。
- 无法确认许可证的代码或素材。
- 长期无人维护且存在明显安全风险的服务。
- 需要把 NAS 管理权限交给第三方 Web 应用的项目。
- 与 Observatory / Lab 主题没有关系的热门项目堆叠。

### 7.4 评审标准

进入第三步前必须确认：

- 上游仓库和许可证。
- 最近维护和安全状态。
- 本站新增价值。
- 更新和退出策略。
- 依赖、包体和运行成本。
- iframe、sandbox、独立路由或独立服务的隔离方式。

### 7.5 与全局架构的关系

Open Lab 不获得额外权限。它仍遵守匿名只读、数据脱敏和独立部署原则。

可能需要的公共能力：

- 第三方许可证清单。
- 上游版本记录。
- sandbox / iframe 安全策略。
- 独立 bundle 或独立服务边界。
- 项目停用开关。

## 8. 板块之间的共享与隔离

### 8.1 可以共享

- Lab 导航和返回路径。
- 页面运行状态语义。
- 数据来源、更新时间和 stale 表达。
- reduced-motion 与低性能检测。
- 错误边界和功能开关。
- Data Gateway 客户端。
- 时间、单位和格式工具。
- 基础安全与隐私规则。

### 8.2 不应强制共享

- 页面模板。
- 主舞台布局。
- 视觉隐喻。
- 参数控制形式。
- 动画框架和渲染技术。
- 项目内容 schema。
- 项目自己的组件层级。

公共层解决平台问题，不负责把所有项目做成相同外观。

## 9. 首版建设原则

Lab 不采用“选择一个板块，从零做到完成，再开始下一个板块”的纵向开发方式。

这种方式会让先完成的板块过早决定全局语言，后续板块只能被迫套用，或者分别发展成互不协调的独立网站。

首版采用横向分层建设：

> 先完成全局框架，再同时建立全部确定板块的最小骨架；随后统一交互、状态和数据运行基础；框架验收后，才进入各板块的具体内容工程。

### 9.1 第一层：Lab 全局框架

- Lab 首页、总导航和返回 Observatory 的路径。
- 全部确定板块的路由与信息层级。
- 页面容器、加载边界、错误边界和项目生命周期。
- 桌面端与移动端的整体布局关系。
- 静态主站与动态 Lab 的视觉过渡。

本层不制作完整实验，只验证 Lab 是否作为一个整体成立。

### 9.2 第二层：板块最小骨架

为所有确定板块同时建立最小结构：

- 板块入口、定位说明与内容边界。
- 板块内的项目承载方式。
- 允许的动态级别、数据依赖和技术自由。
- 返回、空状态、未开放状态和基础降级。

此时可以使用占位内容或脱敏样本，但不能让某一个板块提前进入完整内容开发。

### 9.3 第三层：统一交互与视觉运行规则

- 进入、离开和板块切换体验。
- 动态项目的启动、暂停、重置、销毁和资源释放。
- loading、empty、error、stale 与 unsupported 状态。
- 键盘、触控、移动端和 reduced-motion。
- Canvas、SVG、WebGL 与普通页面之间的切换规则。
- 高成本动态和后台标签页的降级方式。

统一的是运行与状态语言，不是具体项目的页面模板或视觉隐喻。

### 9.4 第四层：公共数据与运行基础

- Data Gateway 客户端与脱敏样本数据。
- 周期快照、更新时间和 stale 判断。
- 时间、单位、来源和 schema 版本表达。
- 项目级错误边界、功能开关和性能检测。
- 本地 mock、NAS 数据与未来 Cloud 数据的切换方式。

### 9.5 第五层：具体内容工程

前四层通过整体验收后，才为各板块选择具体项目、数据和内容。

内容工程仍可分批实现，但必须建立在已经协调的全局框架和板块骨架之上，不能从项目内部反向定义整个 Lab。

## 10. 首版框架验收标准

进入第三步具体内容工程前，第二部分应满足：

- Lab 首页和全局导航成立。
- 所有确定板块都有可比较的最小骨架。
- 不依靠完整内容也能判断板块之间的层级与协调关系。
- 各板块共享的状态、返回、加载和降级语言一致。
- 各板块的视觉自由边界明确，但没有被统一模板同质化。
- 动态运行时可以按项目加载、暂停和销毁。
- NAS 数据可以通过 mock 快照验证 loading、正常、过期和失败状态。
- 桌面端、移动端、键盘与 reduced-motion 行为有统一基线。
- Observatory 不加载 Lab 的大型运行依赖。

## 11. 进入第三步前需要确认

1. 最终确定哪些板块进入 Lab 的长期结构。
2. `Open Lab` 是可见板块，还是开源项目的来源与评审规则。
3. Lab 首页是否只显示已有内容，还是可以显示已确定但尚未填充的板块骨架。
4. 板块名称使用英文主名，还是增加中文副标题。
5. NAS Observatory 是否允许显示人工设置的服务别名，还是只显示系统类别。
6. 全局框架、板块骨架、统一交互和公共数据基础是否已经完成联合验收。
7. 第一批内容应如何在各板块之间形成整体节奏，而不是分别独立立项。

以上确认后，第三步才制定整体内容组合，并分别建立具体项目工程文档。
