## 不是所有服务都在同一层

NAS 上跑着很多东西时，最容易出现一种错觉：所有服务都只是端口列表。

Gitea 是一个端口，金融数据日报是一个端口，远程桌面是一个端口，导航页是一个端口，课程站、地图海报、知识服务、监控脚本也都可以被写进同一张表里。这样整理很方便，但它会遮住一个更重要的问题：这些东西其实不在同一层。

有些服务是底座，有些是入口，有些是日常动作，有些是状态反馈。它们都属于个人系统，但承担的角色完全不同。如果把它们全部看成“我装了什么”，系统就会变成一堆松散工具；如果按层次整理，它们才会开始像一个能长期维护的生活基础设施。

## 底层：存储与运行环境

最底层不是某个具体应用，而是让应用能活着的环境。

NAS、磁盘、Docker、systemd、网络、端口、权限、代理，这些东西平时很少被看见，但它们决定了上层服务能不能稳定运行。磁盘满了，容器会出问题；DNS 解析坏了，反向代理会失败；systemd 残留服务失控，CPU 会被拖高；端口和网络混乱，后面所有工具都会变得难以维护。

这一层不需要每天被欣赏，但必须足够可靠。它像地基：不负责表达什么，却决定整个系统能不能站住。

## 中层：入口与日常动作

中间层负责把系统变得可抵达。

私有 Git 服务让代码、配置和站点内容有地方沉淀；导航页让越来越多的服务不再散落在记忆里；远程桌面让某些必须带界面的操作可以被远程完成；运维面板降低了查看状态和处理服务的成本；知识服务和 API 入口让 Agent 或脚本能读取已经整理过的信息。

这些东西不一定是最终内容，但它们改变了日常动作的路径。以前需要靠记忆、靠临时命令、靠到处翻文件的事情，会慢慢变成一个入口、一个脚本、一个固定位置。

这也是个人系统真正开始成形的地方：不是因为服务数量增加，而是因为重复动作有了稳定路径。

## 上层：观察与反馈

更上层的东西开始把状态推回给我。

金融数据日报、全球资讯速递、用量日报、CPU 异常检测、关注列表监控，这些东西不是为了让系统看起来更复杂，而是让某些变化不用靠我主动想起来才会被发现。

这层最重要的不是自动化，而是反馈。一个数据日报如果没人看，就只是噪音；一个监控如果总是误报，就会被忽略；一个定时任务如果只暴露中间过程，也会消耗注意力。真正有用的反馈应该足够少、足够清楚，并且能让我做出下一步判断。

所以我更愿意把这些任务看成“观察装置”，而不是 dashboard。它们不需要把一切实时展示出来，只需要在关键时候把信号递回来。

## 边界：系统不是站点身份

这些系统很重要，但它们不是 Ayue Observatory 的全部身份。

NAS、自托管、远程桌面、定时任务、Agent、金融数据和监控脚本，都只是个人观测站里的一个切面。它们服务于学习、项目、记录、复盘和长期观察，而不是反过来把这个站定义成基础设施展示页。

这条边界需要反复提醒自己。因为系统越搭越多，很容易产生一种冲动：把首页做成状态面板，把内容变成服务清单，把每一个工具都展示出来。但那样会让 Observatory 失去中心。

OBS-0003 的作用，就是把系统放回它应该在的位置：它是底层支撑，是日常路径，是反馈机制，也是值得记录的生活基础设施。但它不是终点。

## 当前结论

个人系统不是一组端口，也不是一张服务清单。

它更像几层互相支撑的结构：底层让服务活着，中层让动作可抵达，上层把状态反馈回来。真正需要长期维护的，不只是某个应用本身，而是这些层之间的关系。

只要这套关系还清楚，系统就不会只是越堆越多的工具。它会慢慢变成一种更稳定的数字生活环境。
