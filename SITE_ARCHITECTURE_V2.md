# Ayue Observatory Site Architecture V2

> 阶段：第一步／全局网页架构。
>
> 状态：第一部分已确认。本文档定义静态 Observatory 与动态 Lab 共存时的全局工程边界，并接收第二步板块设计和第三步内容工程可能提出的平台需求。

## 1. 架构目标

Ayue Observatory V2 由两个性质不同、但属于同一站点的空间组成：

- **Observatory**：静态、可阅读、可归档、低运行依赖。
- **Lab**：动态、可操作、可定期获取数据、允许高级前端技术。

V2 不是把整个网站升级为动态应用，而是在保留静态主站可靠性的前提下，为 Lab 建立受控的动态运行能力。

核心原则：

> 静态主站不依赖 Lab 才能成立；Lab 可以依赖平台能力运行，但不能把故障、负载和安全风险扩散到主站。

## 2. 全局站点分层

### 2.1 内容层：Observatory

现有 Astro 静态站点继续负责：

- 首页、Archive、Projects、Essays、About。
- Archive Object 与长文阅读。
- RSS、sitemap、canonical 和基础 SEO。
- Lab 的低调入口与必要说明。

Observatory 保持静态生成。即使 NAS、Cloud、数据接口或 Lab 全部不可用，现有页面仍应正常打开。

### 2.2 体验层：Lab Frontend

Lab Frontend 负责：

- Canvas、SVG、WebGL 和复杂动效。
- 参数控制、时间回放、空间导航和状态反馈。
- 周期数据读取与客户端可视化。
- 依赖浏览器能力的交互项目。
- 对数据过期、接口失败和设备不支持的明确降级。

Lab 可以使用独立布局和按项目加载的客户端代码，不要求沿用 Archive 页面模板。

### 2.3 数据层：Lab Data Gateway

Data Gateway 是浏览器与 NAS / Cloud 之间的只读边界，负责：

- 接收经过允许的数据快照。
- 删除敏感字段并输出稳定的数据结构。
- 缓存最后一次成功结果。
- 提供更新时间、来源状态和数据版本。
- 限制请求频率、响应大小和公开范围。

浏览器不得直接访问 DSM、Docker Socket、SSH、数据库管理端口或具备管理权限的内部 API。

### 2.4 采集层：Collectors

Collectors 在 NAS 或受控 Cloud 环境运行，负责：

- 按计划读取允许的数据源。
- 将原始信息转换为最小必要指标。
- 生成脱敏快照。
- 写入 Data Gateway、对象存储或静态数据目录。
- 记录采集成功、失败和最后更新时间。

采集器与公开接口分离。采集凭证只存在于服务端。

## 3. 路由边界

建议全局路由：

```text
/                         Observatory
/archive/*                Archive Objects
/projects
/essays
/about
/lab                      Lab 入口
/lab/<section>            次级板块
/lab/<section>/<project>  具体动态项目
/api/lab/*                只读数据接口（如确有需要）
```

原则：

- `/lab` 属于同一品牌，但进入后使用 Lab 自己的信息架构。
- 每个具体项目可以拥有独立页面结构。
- API 路由与页面路由分开。
- 不把管理接口、采集任务或内部诊断暴露在公开路由中。
- Archive 与 Lab 通过普通链接关联，不建立强制一一对应关系。

## 4. 导航与视觉边界

### 4.1 主站入口

- Lab 可以进入主导航末位。
- Lab 不在 Observatory 首页占据大型宣传区域。
- 主站最多提供一条克制的当前 Lab 项目入口。
- Lab 故障时不影响主导航和主站页面。

### 4.2 Lab 内部导航

- Lab 使用轻量专属栏，提供返回 Lab 与返回 Observatory 的路径。
- 不强制复用完整主站 Header。
- 每个项目必须让用户知道当前所属板块、运行状态和离开方式。

### 4.3 视觉关系

共同保留：

- Ayue Observatory 品牌归属。
- 暗色、低饱和和清晰的文字层级。
- 键盘 focus、移动端降级和 reduced motion。
- 对状态、来源、时间和限制的诚实表达。

Lab 可以突破：

- Archive 的阅读宽度与卡片结构。
- 静态页面的动效强度。
- 单一排版节奏。
- Canvas、SVG、WebGL、图表和实时反馈限制。

## 5. 运行时策略

### 5.1 默认模式

V2 默认采用“静态外壳 + 按项目加载动态运行时”：

- Astro 继续生成主站和 Lab 入口页面。
- Lab 项目只在自己的路由加载客户端代码。
- 项目之间不共享不必要的大型依赖。
- 未访问 Lab 时，不下载 Lab 的可视化运行时。

### 5.2 动态级别

平台预留三种动态级别：

1. **纯客户端动态**：无需服务端，输入与计算在浏览器完成。
2. **周期数据动态**：读取分钟、小时或日级更新的只读快照。
3. **受控交互服务**：调用有明确输入范围、资源限制和权限边界的服务端任务。

第三种不默认开放。每个项目必须在第三步工程文档中单独说明安全和资源限制。

### 5.3 不采用的默认模式

- 不把整个主站改成长期运行的 SPA。
- 不要求 Observatory 页面依赖数据库或在线 API。
- 不建立浏览器到 NAS 管理面的直连。
- 不允许公开执行任意用户代码或系统命令。

## 6. 数据更新模型

### 6.1 周期快照优先

NAS / Cloud 数据默认使用周期快照，而不追求秒级实时：

- 公开 Lab 项目的默认更新周期：每小时。
- 低频趋势和容量数据：每小时或每天。
- 历史回放：按日归档或聚合。
- 如果项目、采集器和数据接口都部署在 NAS 本机，可以按项目评估真正实时或高频更新；启用前必须确认采集负载、渲染成本、安全边界和故障隔离。

具体频率由第二步板块和第三步项目决定，但必须评估采集负载和公开价值。

### 6.2 标准快照元信息

所有公开数据至少包含：

```json
{
  "schemaVersion": "1",
  "generatedAt": "2026-07-13T00:00:00Z",
  "sourceStatus": "ok",
  "staleAfterSeconds": 900,
  "data": {}
}
```

页面不得把“最后一次数据”伪装成当前实时状态。

### 6.3 失败与过期

- 接口失败时优先展示最后成功快照。
- 超过 `staleAfterSeconds` 后显示明确的 `STALE` 状态。
- 无历史快照时显示可理解的无数据状态。
- 动态数据失败不能导致导航、说明和返回路径消失。
- 项目应区分“数据为空”“采集失败”“数据过期”和“浏览器不支持”。

## 7. 公共数据契约

第一步只规定公共元信息，不预先规定所有项目的业务字段。

每个第二步板块可以提出板块级数据约定；每个第三步项目可以定义自己的 `data` 结构。新结构必须：

- 使用版本号。
- 有明确单位和时间范围。
- 避免依赖原始内部名称。
- 允许字段缺失和向后兼容。
- 限制单次响应大小。
- 提供可用于本地开发的脱敏样本。

## 8. 安全与隐私边界

### 8.1 默认禁止公开

- 用户名、账户 ID 和访问令牌。
- 文件名、完整路径和私人目录结构。
- 内网 IP、MAC、设备序列号和精确地理位置。
- 容器环境变量、挂载密钥和完整日志。
- DSM、SSH、Docker Socket 或数据库管理能力。
- 能推断个人活动细节的高精度时间记录。

### 8.2 允许公开的典型数据

- 聚合后的容量比例和变化趋势。
- 经别名处理的服务分类与状态。
- 平滑、降采样后的负载和温度区间。
- 不含私人名称的事件计数。
- 人工确认可公开的项目结果与实验数据。

### 8.3 写操作

Lab 默认只读。任何写操作、任务触发或设备控制都必须：

- 在第三步单独立项。
- 采用最小权限服务。
- 限制允许的动作、参数和频率。
- 不复用 NAS 管理凭证。
- 具备审计、超时、失败回滚和紧急关闭方式。

## 9. 性能与资源预算

全局要求：

- Observatory 的现有静态性能不因 Lab 依赖而下降。
- Lab 代码按路由分割，不进入主站公共 bundle。
- 动态项目必须提供加载、暂停和销毁生命周期。
- 离开项目页面后停止动画循环、轮询和设备监听。
- 后台标签页降低刷新频率或暂停高成本渲染。
- 移动端提供较低采样、较少粒子或静态替代方案。
- 大型历史数据采用聚合、分页或时间窗口加载。

具体帧率、包体和内存预算由第三步项目按技术类型确定。

## 10. 可访问性与控制权

动态技术展示仍需保证用户控制权：

- 支持 `prefers-reduced-motion`。
- 高强度动画必须有暂停或降低强度方式。
- 声音默认不自动播放。
- 关键操作可通过键盘完成。
- 不依赖颜色单独表达状态。
- Canvas / WebGL 项目提供文字说明或状态摘要。
- 不支持的设备显示降级说明，而不是空白。
- 实验必须提供重置方式。

## 11. 部署拓扑

### 11.1 本地开发

- 当前项目目录继续作为唯一源码目录。
- Observatory 与 Lab 页面由同一开发入口访问。
- 动态数据使用脱敏样本或开发接口。
- 本地开发不要求连接真实 NAS 管理服务。

### 11.2 NAS 部署

NAS 可以承担：

- 静态站点文件托管。
- Collectors 计划任务。
- 快照存储和只读 Data Gateway。
- 必要的受控交互服务。

这些职责应拆分为不同进程或容器，不能让公开 Web 服务直接拥有 NAS 管理权限。

### 11.3 Cloud / 公网

未来公网部署可以将：

- 静态页面放在 CDN / 静态托管。
- 公开快照放在缓存或对象存储。
- Data Gateway 放在反向代理之后。
- NAS 通过主动推送或受控隧道更新数据。

不要求公网直接反向访问 NAS 内部管理面。

## 12. 观测与故障隔离

Lab 平台需要记录：

- 采集任务最后成功时间。
- 数据接口成功率和响应时间。
- 快照版本与大小。
- 前端项目的基础错误信息。

但平台自身的运维信息不自动成为公开 Lab 内容。

故障隔离原则：

- Collector 失败不影响 Data Gateway 提供最后快照。
- Data Gateway 失败不影响 Observatory。
- 单个 Lab 项目失败不影响 Lab 入口和其他项目。
- 高负载项目可以被单独禁用。

## 13. 第二步可能提出的架构需求

以下是次级板块设计可能向第一步提出的需求接口，不代表板块已经敲定。

### 13.1 纯动态前端板块

可能需要：

- Canvas / SVG / WebGL 运行环境。
- 资源和着色器按项目加载。
- 全屏实验模式。
- 截图、录制或预设参数分享。
- reduced-motion 与低性能设备降级。

第一步当前已支持前四类中的运行与隔离原则；截图、录制和分享需要第二步确认后再扩展。

### 13.2 NAS 数据展示板块

可能需要：

- 周期快照与历史聚合。
- 数据过期状态。
- 服务、容量、温度、负载和事件的脱敏标识。
- 时间范围切换和回放。
- 多数据源 schema 版本管理。

第一步已预留 Collectors、Data Gateway、缓存和快照元信息。

### 13.3 NAS / Cloud 交互项目板块

可能需要：

- 登录或访问控制。
- 受控任务队列。
- 任务进度、超时和取消。
- 资源配额与频率限制。
- 审计和紧急关闭。

第一步不默认启用这些能力。第二步若确认需要，必须回到本文件补充身份、授权和写操作架构。

### 13.4 开源项目衍生板块

可能需要：

- 第三方许可证记录。
- 上游版本和安全更新跟踪。
- 独立构建或 iframe / sandbox 隔离。
- 与主站数据和账户完全隔离。

第一步允许按项目独立隔离，但具体集成方式必须在第三步评审。

## 14. 第三步内容工程可能提出的公共需求

具体项目可能需要以下公共能力：

- 数据样本与 mock 模式。
- 统一的加载、空数据、失败和过期状态语义。
- 项目级功能开关。
- 项目级资源预算和性能测量。
- 统一时间、单位和数据来源标记。
- 客户端错误边界。
- 静态说明与动态舞台共存。
- Archive 关联链接。

这些能力应优先作为轻量公共约定，而不是强迫所有项目使用同一页面模板。

## 15. 第一部分当前决策

已经确定：

- Observatory 保持静态独立。
- Lab 是动态技术层。
- Lab 允许高级前端技术和周期更新的真实数据。
- 浏览器不直接访问 NAS 管理面。
- 周期快照优先于秒级实时。
- Lab 默认只读。
- 每个大项目可以独立设计。
- 公共能力用于安全、数据、状态和运行边界，不用于统一视觉模板。

## 16. 第一部分确认结论

- Lab 进入主导航末位，保持低于 Archive 的视觉权重。
- 首版不提供服务端写操作，所有公开项目保持匿名只读。
- 数据更新频率由项目定义，未单独说明时默认每小时；仅当项目部署在 NAS 本机并完成资源与安全评估时，才考虑真正实时或高频更新。
- 首版开始保存历史数据，但公开和长期保存的内容以降采样聚合为主。
- 首版公开项目不要求登录；私人数据和私人项目不进入公开 Lab。
- WebGL 是允许的正式能力，但项目必须提供不支持时的降级方案。
- 接受静态页面、Data Gateway 和 Collectors 分开部署。

第一部分据此确认完成，第二部分可以在这些边界内定义具体次级板块。
